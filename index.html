How it works

Deterministic computational pipeline

Your AI decision graph enters as input. A verifiable, reproducible audit bundle comes out. Every time, identically.

Immutable Graph Container

Your evidence graph is encapsulated in a structurally immutable COAContainer. No mutations. No side effects.

Canonical Graph Hash

Deterministic node ordering, edge ordering, identifier normalization. SHA-256 hash computed over the canonical form.

Context Extraction

Bounded subgraph extraction with a focus node, depth parameter, and versioned policy layer. Fully deterministic traversal.

Rule Engine

Structural conflict detection, evidential gap analysis. Fixed priority rule evaluation with deterministic output ordering.

Explainability Score

Bounded σ ∈ [0.0, 1.0] computed from evidence confidence, conflict penalty, and gap penalty. IEEE 754 deterministic rounding.

Verifiable Bundle

TEXT, REPORT, JSON, and PROMPT artifacts. Bundle hash enables independent third-party re-execution and byte-level verification.

Graph format

Build your first decision graph

The engine accepts a directed graph of typed nodes and typed edges. Below is the complete type system, the JSON schema, and two runnable examples. Every example on this page is a valid payload for POST /v1/explain.

Type system
Six node types and six edge types. The engine rejects any value outside this set.
Node types · 6
DECISION
The outcome being explained. The engine focuses analysis on a DECISION node by default.
CLAIM
An intermediate statement the decision depends on. Claims can be supported or contradicted.
EVIDENCE
A fact or signal that supports or contradicts claims. Typically carries a confidence value.
ASSUMPTION
An unverified premise required for the decision. Flagged separately from evidence.
SOURCE
Origin of evidence — a document, registry, API, or database reference.
RULE
A named policy or model rule referenced by claims. Used for traceability.
Edge types · 6
SUPPORTS
Source node positively backs the target. Typical: EVIDENCE → CLAIM.
CONTRADICTS
Source node opposes the target. Detected by the rule engine as a conflict flag.
DEPENDS_ON
Source node requires the target to be valid. Typical: DECISION → CLAIM.
DERIVES
Source node is computed or derived from the target.
REFERS_TO
Weak reference link. Preserved in the graph but does not affect scoring by default.
HAS_SOURCE
Binds an EVIDENCE node to its SOURCE node for provenance tracking.
JSON schema
Required and optional fields. Unknown fields are preserved but do not affect the canonical graph hash.
Request body · application/json 
{
  "graph": {                             // required
    "version": "1.0",                     // optional, defaults to "1.0"
    "nodes": [                            // required, non-empty
      {
        "id":    "n1",                     // required, unique within graph
        "type":  "DECISION",               // required, one of 6 node types
        "label": "Human-readable label",   // required
        "data":  { },                      // optional, arbitrary payload
        "meta":  {
          "confidence": 0.85,           // optional, range [0, 1]
          "tags":       ["finance"],      // optional
          "sourceId":   "s1"             // optional, for EVIDENCE nodes
        }
      }
    ],
    "edges": [                            // required
      {
        "id":   "e1",                      // required, unique within graph
        "type": "SUPPORTS",                // required, one of 6 edge types
        "from": "n2",                      // required, must match a node id
        "to":   "n1",                      // required, must match a node id
        "meta": { "weight": 0.7 }          // optional, range [0, 1]
      }
    ]
  },
  "context": {                           // optional, all fields have defaults
    "layerSet":            "audit",       // audit | human | risk | dev
    "depth":               3,             // 1..10, default 3
    "confidenceThreshold": 0.6,           // 0..1, default 0.6
    "focusNodeId":         "n1"           // default = first DECISION node
  }
}

Built for compliance

Why deterministic matters

Stochastic explanations are hard to audit — identical inputs can produce different outputs. Our pipeline is bit-reproducible by design.

EU AI Act Article 13

Article 13 requires high-risk AI systems to provide transparent, interpretable outputs. Our deterministic pipeline is designed to support these requirements through cryptographic proof chains rather than statistical approximations.

For your auditor They can re-run the computation on their own machine and get byte-identical results. No reliance on our servers, no black box.

Bit-Reproducibility

Identical inputs always produce byte-identical outputs. Across platforms, across time. Verified by SHA-256 bundle hash.

In plain language Think of it as a notary seal on an AI decision. If anything about the explanation changes, the seal no longer matches.

No LLM Dependency

Zero stochastic components. No temperature, no sampling, no prompt sensitivity. Pure deterministic graph computation.

Why it matters An LLM asked the same question twice can give different explanations. The engine gives the same answer every time — so auditors can verify it.

Third-Party Verification

Any auditor can re-execute the pipeline with the same inputs and verify byte-level output identity — designed for independent audit workflows.

What this enables The auditor does not need access to your AI model. They only need the graph, the context parameters, and the proof. Provenance is portable.
